▶ NRQL Writing Tips — Get Better Results
SINCE 1 hour ago — Always set time range explicitly. Default (no SINCE) = last 3 hours. Options: 5 minutes ago live, 1 day ago trends, 1 week ago SLO.
TIMESERIES AUTO — Converts any query to a trend line chart. Remove TIMESERIES entirely for a single billboard/table value.
COMPARE WITH 1 day ago — Overlays the same period from yesterday as a faded baseline. Use 1 week ago for weekly SLO regression reviews.
FACET hostname — Group results by dimension. Common facets: hostname, appName, request.uri, message, db.sql.table.
percentile(duration, 95) — Always use p95 over average(). Averages mask tail latency — the 5% worst requests are what users complain about.
LIMIT MAX — Returns up to 2000 rows. Default LIMIT is 10. Always add for FACET queries where you need all groups, not just top 10.
rate(count(*), 1 minute) — Converts raw event count to a per-minute rate. Use for RPM, errors/min, evictions/min across any time window.
WHERE error IS TRUE — Filters errored transactions only. Combine with percentage(count(*), WHERE error IS TRUE) for error rate %.
FACET CASES(WHERE name LIKE '%checkout%' AS 'Checkout') — Groups raw transaction names into readable buckets. Used in Magento page type breakdown query.
WHERE transactionType = 'Web' — Excludes background jobs (cron, queue consumers) from APM queries. Remove to include non-web transactions.
filter(count(*), WHERE level = 'ERROR') — Inline conditional aggregation. Lets you count multiple conditions in a single query row (e.g. errors + criticals side by side).
convert(duration, unit, 'ms') — Converts metric duration from seconds to milliseconds inline. Needed for apm.service.datastore.operation.duration metrics.
SINCE today — Shortcut for midnight to now. Also: SINCE this week, SINCE this month. Good for daily/weekly dashboards.
WITH ... AS alias — Define constants at query top. Example: WITH 0.8 AS warnSec then use warnSec in WHERE / IF() — cleaner thresholds.
uniqueCount(session) — Counts distinct sessions/users. Use on PageView for live visitor counts. Different from count(*) which counts page hits.
latest(message) — Returns the most recent value of a field. Useful to show the last log message or last error text per hostname.
RLIKE r'/path/.*' — Regex match in WHERE. Use NOT RLIKE r'/.*\.(ico|css|js)' to exclude static asset requests from APM queries.
ORDER BY count(*) DESC — Explicit sort on FACET results. Default sort order is unspecified — always add ORDER BY for ranked tables (top errors, top IPs).
Scope always: Add WHERE appName = 'YOUR-APP-NAME' to every query — without it you query across all apps in your NR account, mixing unrelated data.
Adobe: Use entity.name (not appName) in FROM Log queries. On shared NR instances appName can mismatch; entity.name is stable.

💡 WHY IT MATTERS
APM reveals which endpoints are slow, how many DB calls each makes, and where errors originate — the first stop in any perf investigation
1
Slowest URLs — p95 Response Time
PERFORMANCEMAGENTO
STANDARDADOBE
USE WHENCustomers report slow pages. Find which specific URLs have worst 95th percentile response time — surfaces outliers that averages hide.
FROM Transaction SELECT percentile(duration, 95) AS 'TTFB p95 (s)' WHERE appName = 'YOUR-APP-NAME' AND transactionType = 'Web' FACET request.uri LIMIT 20 SINCE 1 hour ago ORDER BY percentile(duration, 95) DESC
✓ TIPReplace YOUR-APP-NAME with your APM app name. For Adobe Commerce, use the entity.name shown in your NR APM list.
2
Transaction Health Board — 🟢🟡🔴 Status per Endpoint
PERFORMANCEBUSINESS
STANDARDADOBE
USE WHENYou want a single dashboard showing health of every endpoint. Good for NOC screens and daily standups. Green = fast + low errors, Yellow = warning, Red = action needed.
WITH 0.001 AS warnDur, 0.004 AS critDur, 3 AS warnErr, 10 AS critErr SELECT average(duration) AS 'Avg Response (s)', max(duration) AS 'Max Response (s)', count(*) AS 'Total Calls', rate(count(*), 1 minute) AS 'Calls/min', percentage(count(*), WHERE error IS TRUE) AS '% Errors', if(average(duration) < warnDur AND percentage(count(*), WHERE error IS TRUE) < warnErr, '🟢', if(average(duration) < critDur AND percentage(count(*), WHERE error IS TRUE) < critErr, '🟡','🔴')) AS 'Health' FROM Transaction WHERE transactionType = 'Web' AND request.uri NOT RLIKE r'/.*\.(ico|css|js)' FACET appName, request.uri AS 'Endpoint' SINCE 1 day ago LIMIT MAX
3
N+1 Query Detection — DB Calls per Transaction
PERFORMANCEMAGENTO
STANDARDADOBE
USE WHENPages are slow but server CPU looks fine. Magento is notorious for N+1 queries. This buckets transactions into Good/Warning/Poor based on DB call count.
SELECT percentage(count(*), WHERE databaseCallCount < 120) AS '🟢 Good < 120 calls', percentage(count(*), WHERE databaseCallCount >= 120 AND databaseCallCount < 250) AS '🟡 Warning < 250', percentage(count(*), WHERE databaseCallCount >= 250) AS '🔴 Poor > 250' FROM Transaction WHERE appName = 'YOUR-APP-NAME' AND transactionType = 'Web' SINCE 1 week ago COMPARE WITH 1 week ago
⚠️ MAGENTO SPECIFICThresholds 120/250 are tuned for Magento 2. Adjust if your baseline differs. COMPARE WITH shows week-over-week regression.
4
Response Time Quality — SLO Tracking (Good/Needs Work/Poor)
PERFORMANCESLO
STANDARDADOBE
USE WHENTracking weekly SLO targets. Are most transactions fast? Use this in weekly performance reviews.
SELECT percentage(count(*), WHERE duration < 0.8) AS '🟢 Good < 0.8s', percentage(count(*), WHERE duration >= 0.8 AND duration < 1.5) AS '🟡 Needs Work', percentage(count(*), WHERE duration >= 1.5) AS '🔴 Poor > 1.5s' FROM Transaction WHERE appName = 'YOUR-APP-NAME' AND transactionType = 'Web' FACET weekOf(timestamp) SINCE 1 week ago COMPARE WITH 1 week ago

Slowest Magento Transactions — p95 + DB Context
PERFORMANCEMAGENTODB
STANDARDADOBE
USE WHENCombining TTFB with DB call count and DB time to directly tie frontend slowness to database load. Shows which transactions need both code and DB optimization.
FROM Transaction SELECT percentile(duration, 95) AS 'TTFB (p95)', average(databaseCallCount) AS 'DB Calls', average(databaseDuration) AS 'DB Time (ms)' WHERE appName = 'YOUR-FRONTEND-APP-NAME' AND transactionType = 'Web' FACET name LIMIT 20 SINCE 1 hour ago ORDER BY percentile(duration, 95) DESC
✓ TIPHigh DB Calls + high TTFB p95 = N+1 problem. High DB Time + low DB Calls = individual slow query. These require different fixes.

🔄 STANDARD → ADOBE QUERY SUBSTITUTION GUIDE
TRAFFIC ANALYSIS
Standard uses nginx logs:
FROM Log WHERE filePath = '...'
Adobe — use APM transactions:
FROM Transaction SELECT count(*), rate(count(*),1 minute) WHERE entity.name = 'YOUR-APP-NAME' TIMESERIES 1 minute
ERROR LOGS
Standard uses appName:
FROM Log WHERE appName = 'YOUR-APP-NAME' AND level = 'CRITICAL'
Adobe — use entity.name:
FROM Log WHERE entity.name = 'YOUR-APP-NAME' AND level = 'CRITICAL' AND newrelic.source = 'logs.APM'
DATABASE SLOW QUERIES
Standard uses entity.guid (infra) or Span:
FROM Metric WHERE entity.guid = '...' SELECT average(mysql.node.slowQueriesPerSecond)
❌ Span & Metric not available on Adobe. Use Transaction:
FROM Transaction SELECT average(databaseDuration) AS 'Avg DB (s)', average(databaseCallCount) AS 'DB Calls' WHERE entity.name = 'YOUR-APP-NAME' FACET request.uri LIMIT 20
N+1 / DB CALL COUNT
Works on both — no change needed:
FROM Transaction SELECT average(databaseCallCount), average(databaseDuration) WHERE entity.name = 'YOUR-APP-NAME' FACET request.uri LIMIT 20
SLOW TRANSACTIONS (p95)
Works on both — swap appName → entity.name:
FROM Transaction SELECT percentile(duration, 95) AS 'p95' WHERE entity.name = 'YOUR-APP-NAME' AND transactionType = 'Web' FACET request.uri ORDER BY percentile(duration,95) DESC
OPEN ALERT INCIDENTS
Works on both — no change needed:
FROM NrAiIncident SELECT count(*) WHERE event = 'open' FACET conditionName, entity.name SINCE 1 day ago

4. Skip Redis, ES, Infra, and PHP-FPM sections entirely — Adobe manages those services.
5. Sections that work fully on Adobe: APM · Errors · Magento M2 · Business · Alerts · Executive.
